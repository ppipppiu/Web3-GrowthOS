# Web3 DApp 市场与竞品研究

## 1. 研究背景

我准备参加 Web3 黑客松，但在确定项目方向时，对目前市场上常见的 DApp 类型、Web3 产品的实际使用场景，以及不同项目的产品价值还不够了解。

因此，本次研究主要围绕以下问题展开：

1. Web3 中常见的 DApp 有哪些类型？
2. 当前黑客松中常见的产品方向是什么？
3. 链上用户分析、用户增长和精准激励是否存在真实需求？
4. 市场上是否已经存在类似产品？
5. 我的项目应该如何与现有产品形成差异化？
6. 黑客松版本应该实现哪些核心功能？

本次研究最终聚焦于以下项目方向：

> 面向 Web3 初创项目的链上用户分析、用户分层与精准激励平台。

项目希望通过分析智能合约和钱包的链上交互数据，识别不同类型的用户，并帮助项目方针对特定人群创建相应的运营和奖励活动。

---

# 2. Web3 DApp 的主要类型

Web3 DApp 并不是单一类型的产品。根据产品服务对象和链上功能，可以大致分为以下几个方向。

## 2.1 DeFi

DeFi，即去中心化金融，是目前最典型的 Web3 产品方向。

常见产品包括：

* Token 兑换
* 去中心化交易所
* 借贷协议
* 稳定币
* 质押
* 收益聚合
* 永续合约
* 期权
* 预测市场
* 链上保险
* 资产管理

DeFi 的核心特点是用户可以直接控制自己的资产，并通过智能合约完成交易、借贷和结算。

这一方向具有较强的 Web3 必要性，但对智能合约安全、金融机制设计和资金风险控制的要求较高。

对于新手黑客松项目，可以考虑实现：

* 模拟交易 DApp
* 简单的链上定投工具
* 钱包资产分析工具
* 交易风险提示
* 链上策略分析工具

不建议新手直接开发完整的 DEX、借贷协议或者复杂衍生品协议。

---

## 2.2 支付与结算

这一类型主要利用稳定币和智能合约完成支付、托管和自动结算。

常见场景包括：

* 跨境支付
* 自由职业者薪酬结算
* 商家收款
* 链上订阅支付
* 多人自动分账
* 托管交易
* 服务完成后的自动结算

典型流程是：

```text
创建订单
→ 锁定资金
→ 完成交付
→ 确认结果
→ 智能合约自动结算
```

这一方向的业务流程清晰，也容易在黑客松中展示智能合约的价值。

---

## 2.3 NFT 与数字所有权

NFT 不仅可以代表数字图片，也可以代表某种权益、身份或者数字资产。

常见应用包括：

* 活动门票
* 游戏道具
* 会员资格
* 数字证书
* 创作者作品
* 成就徽章
* 活动纪念品
* 可验证的学习记录
* 可转让或不可转让的凭证

NFT 项目的核心不应该只是“铸造一个 NFT”，而是需要回答：

1. NFT 代表什么权益？
2. 用户为什么需要持有它？
3. NFT 是否可以解锁具体功能？
4. 所有权是否需要公开验证？

单纯的 NFT Mint 页面已经比较同质化，不适合作为项目的主要创新点。

---

## 2.4 SocialFi 与社区产品

SocialFi 是将社交产品、创作者经济和链上资产结合起来的方向。

常见功能包括：

* 链上社交身份
* 社区积分
* 粉丝会员
* Token-gated 社区
* 创作者收益分配
* 内容所有权
* 链上声誉
* 社交关系图谱

这一类型适合加入：

* 钱包登录
* 链上身份
* 内容确权
* 社区积分
* 声誉系统
* 贡献激励

但 SocialFi 产品通常存在较明显的用户冷启动问题。

因此，黑客松项目更适合服务某一个明确的小型社区，而不是直接开发一个“Web3 版微博”。

---

## 2.5 DAO 与治理工具

DAO 工具主要服务去中心化组织的治理和协作。

常见功能包括：

* 治理提案
* 链上投票
* 多签资金管理
* 贡献记录
* 任务赏金
* 社区预算
* 成员权限管理
* 治理数据分析

可以进一步开发：

* 基于贡献度的投票系统
* 女巫地址识别
* 治理提案自动摘要
* 投票数据可视化
* DAO 成员活跃度分析
* 核心贡献者识别
* 沉默成员召回

这一方向与数据分析和社区运营的结合程度较高。

---

## 2.6 链上身份、声誉与凭证

这一方向主要解决用户如何证明自己的身份、经历或资格。

常见产品包括：

* DID 去中心化身份
* 学习证明
* 链上简历
* 工作经历凭证
* 活动参与证明
* 链上信用评分
* 用户声誉系统
* Soulbound Token
* 隐私证明

核心问题是：

> 用户如何在不完全公开个人信息的情况下，证明自己满足某个条件？

例如：

* 证明参加过某项活动
* 证明具有某种会员资格
* 证明是社区核心贡献者
* 证明信用达到指定标准
* 证明年龄符合要求，但不公开完整身份信息

---

## 2.7 游戏与消费者应用

这一方向包括：

* 链上游戏
* 游戏道具所有权
* 玩家经济系统
* 链上成就系统
* 预测竞猜
* 收藏系统
* 任务和奖励
* 签到和积分

这类项目容易制作可视化 Demo，但不能只停留在简单的签到功能。

更完整的用户闭环应该是：

```text
用户完成行为
→ 系统验证行为
→ 用户获得积分或成就
→ 解锁新的权益
→ 促进用户持续参与
```

---

## 2.8 RWA

RWA，即现实世界资产上链，主要将现实中的资产、收益或所有权映射到链上。

常见方向包括：

* 房地产
* 发票
* 债券
* 黄金
* 碳信用
* 供应链资产
* 商品溯源
* 会员权益

区块链可以负责：

* 资产所有权记录
* 收益分配
* 交易结算
* 状态验证
* 资产流转记录

但是，真正的 RWA 项目通常涉及法律、托管和合规问题。

黑客松中更适合开发原型，不应该直接声称已经解决现实资产的完整合规问题。

---

## 2.9 DePIN

DePIN，即去中心化物理基础设施网络。

常见场景包括：

* 共享存储
* 分布式计算资源
* 网络带宽
* 地图数据
* 传感器数据
* 能源设备
* 物联网设备

基本模式是：

```text
用户贡献现实资源或数据
→ 系统验证贡献
→ 根据贡献发放奖励
```

这个方向具有一定的真实业务价值，但通常需要硬件、外部数据或者复杂的验证系统，因此开发难度较高。

---

## 2.10 Web3 数据分析与增长工具

这一方向主要帮助协议和 DApp 团队理解链上用户行为。

常见功能包括：

* 钱包行为分析
* 链上用户画像
* 用户分层
* 协议增长看板
* 新增用户分析
* 用户活跃分析
* 用户留存分析
* 用户生命周期分析
* 空投用户识别
* 女巫地址识别
* DAO 治理分析
* 营销活动效果评估
* 跨协议用户迁移分析

这一类产品通常采用链上与链下结合的架构：

```text
区块链
→ 提供公开、可验证的行为数据

链下数据系统
→ 完成数据清洗、指标计算、模型分析和可视化

智能合约
→ 负责身份、权限、奖励和策略执行
```

这一方向与数据分析、用户分层、用户增长和运营策略具有较高的结合度。

---

## 2.11 AI Agent 与 Web3

AI Agent 与 Web3 的结合也是目前黑客松中比较常见的方向。

常见场景包括：

* AI 自动执行链上交易
* AI 钱包助手
* AI 分析治理提案
* AI 检测智能合约风险
* AI 进行资产管理
* Agent 之间自动支付
* 自然语言调用多个 Web3 协议
* AI 生成用户运营策略

但一个 AI Web3 项目不能只是接入大语言模型 API。

项目必须明确：

1. AI 负责什么判断？
2. 区块链负责什么可信执行？
3. 为什么不能直接使用普通的 ChatGPT？
4. AI 的建议是否可以转化为实际链上操作？

# 3. 如何判断一个产品是否真的需要 Web3

一个产品是否应该使用区块链，可以通过以下四个问题判断。

## 3.1 是否涉及资产或者权益

例如：

* Token
* NFT
* 钱
* 积分
* 所有权
* 会员资格
* 投票权
* 分红权

当产品涉及可交易或者可验证的资产和权益时，上链通常更加合理。

---

## 3.2 是否需要多方共同信任一套规则

例如：

* 用户
* 商家
* 项目方
* 创作者
* DAO 成员
* 投资人

这些参与者之间不一定完全信任，但需要共同认可某套结算、奖励或者治理规则。

智能合约可以让这些规则公开并自动执行。

---

## 3.3 记录是否需要公开验证和防止随意修改

例如：

* 捐款去向
* 投票结果
* 奖励发放
* 交易记录
* 社区贡献
* 数字证书
* 资产归属

如果记录需要被多个参与者共同验证，区块链可能具有明显价值。

---

## 3.4 用户是否需要自己控制身份和资产

如果用户完全依赖中心化平台，平台可以随时冻结、删除或修改用户的权益，那么链上身份和链上所有权可能具有价值。

如果以上四个问题的答案全部为“否”，这个产品通常使用 Web2 技术就可以完成。

# 5. 黑客松项目模式总结

根据以上项目，可以将相关产品分为四种主要模式。

## 5.1 纯数据分析工具

代表项目包括：

* Crossbow
* WEB3 Board
* DAO Hub
* GovLens

基本流程是：

```text
读取链上数据
→ 计算指标
→ 生成看板
```

优点：

* 开发逻辑清晰
* 容易形成可视化 Demo
* 能够展示数据分析能力
* 链上数据来源明确

缺点：

* 容易变成普通 BI 看板
* 产品差异化较弱
* 用户看完数据后仍然不知道下一步应该做什么
* 缺少实际运营执行能力

---

## 5.2 AI 数据洞察工具

代表项目包括：

* Agenta
* MetricsFlow
* TABULA
* DAOscope

基本流程是：

```text
读取数据
→ AI 分析
→ 输出摘要或者建议
```

优点：

* 降低数据分析的使用门槛
* 运营人员不需要编写 SQL
* 可以自动解释复杂指标
* 更适合非技术用户

缺点：

* AI 可能只是重新总结已有数据
* 如果没有后续执行环节，产品价值仍然有限
* AI 生成的建议可能缺少可验证性

更有价值的方式是让 AI 参与：

* 用户筛选
* 用户分层
* 活动方案设计
* 奖励规则推荐
* 实际活动参数配置

---

## 5.3 任务和奖励平台

代表项目包括：

* TwitterCampaign
* Cashback SDK
* GaaS
* Proof-of-Loyalty
* Show-Tie

基本流程是：

```text
项目方创建任务
→ 用户完成行为
→ 系统验证行为
→ 发放奖励
```

优点：

* 业务流程清晰
* 容易形成完整 Demo
* 智能合约价值明显
* 用户和项目方都可以直接参与

缺点：

* 用户分层通常比较简单
* 经常只判断“满足条件”或“不满足条件”
* 可能吸引只为了奖励而来的用户
* 不一定能够产生长期留存

---

## 5.4 分析与运营闭环

代表项目包括：

* VICCI-Web3
* Themis
* Loyalty
* Yam Yam Yam

基本流程是：

```text
分析用户
→ 识别人群
→ 制定策略
→ 发放激励
→ 查看活动结果
```

这种模式不仅展示数据，还能根据分析结果直接执行运营策略。

这一方向与用户增长、数据分析和 Web3 智能合约的结合程度最高，也更适合作为本次黑客松项目的主要方向。

# 6. 正式产品竞品分析

## 6.1 Galxe

### 产品定位

Galxe 是 Web3 用户增长和任务活动平台。

项目方可以创建链上或链下任务，用户完成指定行为后，可以获得：

* NFT
* Token
* 积分
* 身份凭证
* 社区权益

常见任务包括：

* 关注社交账号
* 加入社区
* 连接钱包
* 与智能合约交互
* 持有指定 Token
* 完成特定链上操作

### 主要优势

* 任务创建能力强
* 链上和链下条件验证较成熟
* 奖励发放流程完整
* 已经形成较大的用户和项目生态
* 支持身份和用户评分

### 主要不足

Galxe 更擅长回答：

> 如何创建一个增长任务？

但它不一定能够充分回答：

* 为什么应该选择这些用户？
* 哪类用户最值得激励？
* 哪些用户可能流失？
* 活动是否真的提高了长期留存？
* 不同用户应该获得什么不同的奖励？

### 对项目的启发

本项目不应该直接开发一个简化版 Galxe。

更合理的方向是：

> 在创建任务之前，帮助项目方完成用户分析、用户分层和目标人群筛选。

---

## 6.2 Layer3

### 产品定位

Layer3 是面向 Web3 项目方和普通用户的任务、奖励和增长平台。

项目方可以创建链上任务，用户完成任务后获得奖励。

### 主要优势

* 任务流程比较成熟
* 覆盖多条区块链
* 支持链上行为追踪
* 可以帮助协议完成用户获取
* 已经积累大量任务和用户行为数据

### 主要不足

任务激励平台容易出现以下问题：

* 用户只为了奖励而参与
* 奖励结束后活跃度下降
* 新增钱包不一定等于真实新增用户
* 难以判断任务带来的长期价值
* 容易产生羊毛用户和机器人用户

### 对项目的启发

可以将 Layer3 理解为增长活动的“执行层”。

本项目更适合作为增长活动的“决策层”：

> Layer3 帮助项目创建任务，本项目帮助项目判断应该给谁创建什么任务。

---

## 6.3 Cookie3

### 产品定位

Cookie3 更接近 Web3 版本的营销分析和归因平台。

它可以将以下行为连接起来：

```text
营销渠道
→ 网站访问
→ 钱包连接
→ 链上交互
→ Token 购买或协议转化
```

主要帮助项目方分析：

* 用户从哪里来
* 哪个营销活动带来更多钱包连接
* 哪个渠道产生实际链上转化
* KOL 活动的效果
* 不同营销渠道的 ROI

### 主要优势

* 强调链上与链下数据结合
* 营销归因能力较强
* 能够连接流量来源和链上转化
* 更接近 Web2 的增长分析体系

### 主要不足

Cookie3 更强调整体渠道和营销效果。

本项目可以进一步关注：

* 用户生命周期
* 用户价值评分
* 用户活跃程度
* 用户流失风险
* 用户分层
* 差异化运营策略
* 活动前后的留存变化

### 对项目的启发

Cookie3 是与本项目定位比较接近的竞品之一。

本项目需要突出：

> 不只是分析用户从哪里来，还要判断用户现在处于什么阶段，以及下一步应该采用什么运营策略。

---

## 6.4 Dune

### 产品定位

Dune 是通用型链上数据查询和分析平台。

用户可以：

* 使用 SQL 查询链上数据
* 创建公开或私有 Dashboard
* 分析协议增长
* 分析用户活跃
* 分析用户留存
* 分析 Token 和交易数据
* 通过 API 获取数据

### 主要优势

* 链上数据覆盖范围广
* 查询能力灵活
* 社区中已经存在大量 Dashboard
* 适合专业分析师和研究人员
* 可以支持自定义指标分析

### 主要不足

* 使用门槛较高
* 通常需要 SQL 能力
* 运营人员不一定能够直接使用
* 主要提供数据和看板
* 不直接负责运营策略执行

### 对项目的启发

本项目不需要与 Dune 比较数据覆盖范围。

更适合的差异化是：

> 将需要 SQL 和数据分析能力的链上分析，封装为运营人员可以直接使用的一键式工具。

## 6.5 Footprint Analytics

### 产品定位

Footprint Analytics 提供：

* 链上数据看板
* 统一数据 API
* 低代码分析工具
* 用户增长分析
* 协议分析
* GameFi 分析
* NFT 分析

用户可以通过拖拽或者低代码方式建立数据看板。

### 主要优势

* 降低了链上数据分析门槛
* 支持多个数据领域
* 可以快速创建 Dashboard
* 覆盖常见增长指标
* 适合项目方和数据分析师

### 主要不足

如果项目只实现以下指标：

* 新增用户
* 活跃用户
* 交易次数
* 钱包余额
* 用户留存
* 交易量趋势

那么与 Footprint 的差异仍然不明显。

### 对项目的启发

本项目应该进一步实现：

```text
用户分层
→ 策略推荐
→ 人群选择
→ 活动执行
→ 效果评估
```

# 8. 市场研究结论

通过黑客松项目和正式产品研究，可以得出以下结论。

## 8.1 这个方向存在真实需求

Galxe、Layer3、Cookie3、Dune 和 Footprint 分别验证了以下市场需求：

* Web3 项目需要创建用户增长活动
* 项目需要验证用户是否完成链上行为
* 项目需要向用户发放链上奖励
* 项目需要分析链上用户和协议数据
* 项目需要评估营销和增长活动的效果

因此，链上用户分析和用户增长不是完全虚构的需求。

---

## 8.2 单纯的数据看板已经比较同质化

市场中已经存在大量：

* 链上数据看板
* 协议分析平台
* 钱包分析工具
* DAO 治理分析工具
* AI 数据问答工具

如果项目只是展示新增钱包、活跃钱包、交易次数和留存率，产品差异化会比较弱。

---

## 8.3 单纯的任务平台也已经较为成熟

Galxe 和 Layer3 已经可以完成：

* 创建任务
* 验证任务
* 发放奖励
* 管理活动
* 追踪任务完成情况

因此，项目不应该只开发一个普通任务平台。

---

## 8.4 当前仍然存在分析和执行之间的空白

数据平台通常能够回答：

> 用户发生了什么行为？

任务平台通常能够回答：

> 如何创建一个奖励活动？

但项目方仍然需要解决：

* 应该选择哪些用户？
* 为什么选择这些用户？
* 不同用户应该获得什么策略？
* 哪些用户可能流失？
* 哪些用户只是为了领取奖励？
* 活动是否真的提高了留存？

因此，本项目可以连接分析工具和任务平台：

```text
数据分析
→ 用户分层
→ 策略推荐
→ 创建活动
→ 发放奖励
→ 效果评估
```

# 10. 黑客松 MVP 设计

黑客松版本不需要开发完整的平台，只需要实现一条清晰、可演示的业务闭环。

建议实现以下四个主要页面。

## 10.1 项目导入页面

用户输入：

* 智能合约地址
* 区块链网络
* 分析时间范围

系统读取：

* 合约交易
* Event 日志
* 钱包地址
* 交互时间
* 交互次数
* 交易金额

基础流程：

```text
输入合约地址
→ 获取链上数据
→ 清洗和聚合钱包行为
→ 生成分析结果
```

---

## 10.2 增长看板页面

展示以下指标：

* 新增钱包数
* 活跃钱包数
* 独立交互钱包数
* 新老钱包比例
* 平均交互次数
* 交易量趋势
* 次日留存率
* 7 日留存率
* 不同时间段的活跃趋势

这些指标用于帮助项目方快速了解整体增长情况。

---

## 10.3 用户分层页面

黑客松版本可以先使用可解释的规则分层，不需要一开始就实现复杂机器学习模型。

| 用户层级   | 示例规则             | 运营含义        |
| ------ | ---------------- | ----------- |
| 新用户    | 首次交互时间不超过 7 天    | 需要引导完成第二次交互 |
| 高价值用户  | 交互频次或金额位于前 20%   | 适合提供高级权益    |
| 活跃用户   | 近 7 天存在多次交互      | 适合推动更深层功能使用 |
| 流失风险用户 | 过去活跃，但近 14 天没有交互 | 适合召回活动      |
| 奖励型用户  | 只参与奖励活动，没有核心交互   | 需要增加领奖门槛    |
| 普通用户   | 存在交互，但活跃和价值一般    | 适合基础任务引导    |

系统需要展示：

* 每类钱包的数量
* 每类钱包的比例
* 具体钱包地址
* 对应行为指标
* 推荐运营策略

---

## 10.4 策略与奖励页面

项目方选择目标人群后，系统推荐相应策略。

### 新用户

策略示例：

* 完成第二次交互后领取徽章
* 提供新用户操作引导
* 设置限时任务
* 引导体验核心功能

### 高价值用户

策略示例：

* 发放高级会员 NFT
* 提供治理权限
* 提供专属活动资格
* 提供手续费优惠

### 活跃用户

策略示例：

* 推动使用更多产品功能
* 设置连续交互任务
* 提供社区贡献奖励
* 引导邀请新用户

### 流失风险用户

策略示例：

* 发放限时召回奖励
* 设置重新激活任务
* 提供回归徽章
* 推荐较低门槛的交互任务

### 奖励型用户

策略示例：

* 必须完成核心业务行为后才能领奖
* 提高奖励领取条件
* 限制重复领取
* 加入持有时间或连续活跃要求

# 13. 项目的差异化

本项目不应该强调：

* 数据比 Dune 更多
* 任务比 Galxe 更丰富
* 奖励比 Layer3 更复杂
* 能够取代成熟商业产品

本项目更合理的差异化是：

> 比 Dune 更容易使用，比 Galxe 更懂用户，比普通 AI 看板更接近实际运营执行。

具体差异包括：

1. 不要求运营人员编写 SQL。
2. 不只是展示数据，还自动完成人群划分。
3. 不只是创建统一任务，而是针对不同用户创建不同策略。
4. 将数据分析结果直接连接到链上奖励活动。
5. 可以继续扩展活动前后的效果评估。
6. 更适合缺少专职数据分析师的小型 Web3 团队。

# 14. 当前项目风险

## 14.1 钱包不等于真实用户

一个真实用户可能拥有多个钱包。

一个钱包也可能属于：

* 机器人
* 团队
* 交易所
* 多签账户
* 智能合约
* 自动化程序

因此，项目中应该使用：

* 活跃钱包
* 新增钱包
* 钱包分层
* 钱包行为

而不是直接将每一个地址称为“真实用户”。

---

## 14.2 链上数据无法覆盖全部用户行为

用户可能在以下环节产生链下行为：

* 浏览页面
* 点击按钮
* 查看活动
* 阅读内容
* 加入社区
* 完成社交任务

这些行为不会自动记录在区块链中。

因此，完整的增长分析可能需要结合：

* 链上交易数据
* 网站埋点数据
* 社交平台数据
* 活动平台数据

黑客松版本可以先聚焦链上行为。

---

## 14.3 激励可能产生羊毛用户

Token、NFT 和积分奖励可能吸引只为了奖励而来的用户。

因此，奖励活动需要考虑：

* 核心行为要求
* 奖励领取门槛
* 持续活跃要求
* 重复领取限制
* 钱包行为质量
* 奖励后的留存情况

---

## 14.4 项目范围容易过大

完整产品可能涉及：

* 多链数据读取
* 钱包画像
* 用户分层
* AI 策略
* 活动创建
* 奖励合约
* A/B 实验
* 因果推断
* 自动化运营
* 反女巫识别

黑客松版本不能一次全部实现。

最合理的 MVP 是：

```text
输入一个合约地址
→ 分析钱包行为
→ 完成规则分层
→ 选择目标人群
→ 创建链上奖励
```

---

# 15. 后续迭代方向

黑客松 MVP 完成后，可以继续扩展以下能力。

## 15.1 用户价值评分

综合以下指标计算用户价值：

* 交互频次
* 交互金额
* 活跃天数
* 使用功能数量
* 最近一次交互时间
* 持续活跃时间

---

## 15.2 用户流失预测

通过历史行为预测用户是否可能停止交互。

可以使用：

* Logistic Regression
* Decision Tree
* Random Forest
* XGBoost
* LightGBM

但需要足够的历史用户数据。

---

## 15.3 因果推断与增量分析

当平台积累足够的活动数据后，可以进一步分析：

* 哪类用户真正受到奖励影响？
* 哪些用户即使不奖励也会继续使用？
* 哪些奖励对哪些用户有效？
* 奖励是否真正提高留存？

可以进一步使用：

* Uplift Modeling
* Causal Forest
* CATE
* A/B Testing
* Heterogeneous Treatment Effect

这一部分更适合产品已经有较多历史数据以后实现，不适合作为最初 MVP 的核心。

# 16. 最终结论

链上用户分析和用户增长属于具有真实需求的 Web3 产品方向。

市场中已经存在：

* Dune 和 Footprint 等链上数据分析平台
* Cookie3 等营销归因平台
* Galxe 和 Layer3 等任务与奖励平台
* 多种 AI 链上数据分析工具

因此，单纯开发数据看板或者任务平台已经缺少明显差异化。

本项目更适合连接数据分析和运营执行：

```text
链上数据采集
→ 钱包行为分析
→ 用户分层
→ 策略推荐
→ 链上奖励
→ 活动效果评估
```

项目的主要差异化是：

> 根据真实链上行为判断应该激励哪些钱包，以及不同钱包应该采用什么运营策略，而不是向所有用户发放相同奖励。

黑客松版本应该控制范围，优先完成以下核心闭环：

```text
输入合约地址
→ 展示增长指标
→ 完成钱包分层
→ 选择目标人群
→ 创建链上奖励活动
```

这个方向能够同时体现：

* Web3 产品理解
* 智能合约开发能力
* 前端可视化能力
* 链上数据分析能力
* 用户增长和运营思维

与单纯的签到 DApp 相比，该项目具有更完整的业务逻辑，也更适合作为黑客松作品和数据分析求职项目。
