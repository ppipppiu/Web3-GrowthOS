## 共同关注的问题

早期 Web3 项目缺少简单、低成本的用户分析工具。

项目方虽然可以获取钱包交易数据，但很难快速完成数据清洗、用户行为分析和用户分层，也难以判断哪些钱包用户更有价值、应该优先运营哪些用户。

团队计划共同完成 Web3-GrowthOS Demo，让 Web3 项目方可以上传交易 CSV 文件，自动获得：

- 数据质量和清洗结果
- 总钱包数、交易数和交易金额等指标
- 钱包级交易行为数据
- 用户分层结果
- 可视化分析 Dashboard

## 目前缺少的角色

目前团队由两名成员组成，已经覆盖 Dev、Data、Research、Product 和基础 Frontend 工作。

暂时缺少：

- UI/UX 设计经验较强的成员
- 熟悉真实链上数据接口或数据索引服务的 Web3 Data Builder
- 熟悉 Web3 项目增长运营的 Ops Builder

这些角色不是 Demo v0.1 的必要条件，但可以在后续 Hackathon 阶段帮助团队完善产品体验、链上数据接入和运营策略。

## 基础协作方式

- 团队沟通：面对面进行日常沟通和问题同步。
- 任务看板：使用 GitHub Projects 或 GitHub Issues 记录待办、进行中和已完成任务。
- 项目资料：统一存放在 Web3-GrowthOS GitHub 仓库中。
- 代码协作：通过 GitHub 提交代码和文档，重要修改在合并前由另一名成员检查。
- 项目记录：由 yi个小小yi 负责记录任务、决策和每周进度。
- 项目推进：由 PLLL 负责确认优先级、跟进进度和组织 Demo 展示。

## 每周同步时间

随时进行同步。

同步内容包括：

1. 本周已经完成的任务
2. 当前遇到的问题
3. 下周需要完成的任务
4. 是否需要调整分工
5. 是否继续进入下一阶段或 Hackathon

如遇到影响项目进度的问题，在群聊中及时同步，不等待周会。

## 任务分配方式

团队根据成员能力进行模块化分工：

- PLLL 主要负责数据分析、后端开发、项目推进和 Demo 展示。
- yi个小小yi 主要负责 Research、产品设计、前端开发和项目记录。
- 前后端接口、功能范围和最终展示由双方共同确认。

每项任务需要明确：

- 负责人
- 预期产出
- 完成时间
- 当前状态

## 决策方式

一般问题由对应模块负责人决定。

涉及产品方向、功能范围、技术路线或工作量变化的重要问题，由两名成员共同讨论并达成一致。

如果短时间内无法达成一致，则优先选择：

1. 能够在本周完成的方案
2. 技术风险较低的方案
3. 更符合 Mini Demo 验收要求的方案
4. 后续方便继续迭代的方案

## 安全边界

团队明确遵守以下安全规则：

- 不在群聊、GitHub 或文档中共享私钥
- 不共享钱包助记词
- 不上传真实账户密码或 API 密钥
- 不提交包含敏感信息的 `.env` 文件
- 测试时使用测试钱包和测试网络
- GitHub 仓库只保存公开代码、测试数据和必要文档
- 如发现密钥误传，立即删除并更换相关密钥
