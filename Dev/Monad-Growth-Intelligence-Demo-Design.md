# Monad Growth Intelligence Demo Design

## 1. 项目概述

### 1.1 项目名称

**Monad Growth Intelligence**

### 1.2 项目定位

Monad Growth Intelligence 是一个面向 Monad 生态 DApp 项目的用户增长与数据智能 Demo。

平台允许项目方上传用户行为数据，自动生成：

* 用户增长数据看板
* 用户分层结果
* 用户运营策略
* 可验证的链上分析凭证

本次 Demo 不尝试建设完整的企业级数据基础设施，而是验证一条最小可行的产品闭环：

```text
接入数据
→ 计算增长指标
→ 展示用户看板
→ 生成运营策略
→ 确认分析结果
→ 将报告哈希写入 Monad
```

---

## 2. 项目背景

Web3 DApp 项目通常可以从区块浏览器或数据平台中查看：

* 钱包数量
* 交易数量
* 交易金额
* 活跃地址数量
* 合约调用记录

但这些数据通常无法直接回答以下增长问题：

* 新用户是否完成了关键行为？
* 用户完成第一次交互后是否继续使用？
* 哪些用户正在流失？
* 哪些用户值得继续激励？
* 哪些用户可能只是为了领取奖励？
* 当前运营活动是否真正提高了留存？
* 下一轮应该优先运营哪些用户？

传统链上数据平台更擅长回答：

> 链上发生了什么？

而 Monad Growth Intelligence 试图进一步回答：

> 为什么会出现这些结果，以及项目方下一步应该运营哪些用户？

---

## 3. 目标用户

本次 Demo 的目标用户是：

* Monad 生态内的早期 DApp 项目方
* 缺少专职数据分析师的小型 Web3 团队
* 负责用户增长和社区运营的成员
* 希望评估用户质量和运营活动效果的项目负责人
* 参加黑客松并需要快速验证用户增长方案的开发团队

本次优先服务交易类 DApp，例如：

* DEX
* Swap 应用
* Meme Token 交易平台
* 链上资产交易平台
* 预测市场
* 游戏资产交易市场

交易类 DApp 的用户行为路径相对清晰，常见流程包括：

```text
首次与合约交互
→ 完成首次交易
→ 完成第二次交易
→ 成为活跃用户
→ 成为高价值用户
```

因此更适合用于第一版增长分析 Demo。

---

## 4. 核心问题

Monad Growth Intelligence 重点解决以下问题：

### 4.1 数据分散

DApp 项目的用户行为数据可能来自：

* 链上交易
* 智能合约事件
* CSV 文件
* 活动记录
* 奖励领取记录

项目方通常需要手动整理数据，才能进行用户分析。

### 4.2 数据无法直接转化为策略

即使项目方已经知道：

* 活跃钱包数量
* 交易次数
* 交易金额
* 用户留存率

仍然可能不知道：

* 应该优先运营哪类用户
* 应该采用什么运营动作
* 是否应该继续发放奖励
* 下一轮活动应该观察什么指标

### 4.3 运营结果缺少验证

传统分析报告通常存储在中心化服务器中，报告内容可能被修改，也无法证明：

* 报告由谁生成
* 报告何时生成
* 报告对应哪份数据
* 当前报告是否为原始版本

因此，本项目将分析报告的哈希写入 Monad，生成可验证的链上分析记录。

---

## 5. Demo 功能组成

本次 Demo 最终由四个核心功能组成：

1. 项目创建与数据接入
2. 数据看板与用户分层
3. AI 用户运营策略
4. 链上分析凭证

---

# 6. 功能一：项目创建与数据接入

## 6.1 功能目标

让项目方可以创建一个增长分析项目，并上传需要分析的用户行为数据。

## 6.2 用户操作

用户进入平台后，可以：

1. 连接钱包
2. 创建分析项目
3. 填写项目基本信息
4. 选择 DApp 类型
5. 上传 CSV 文件
6. 使用 Demo 示例数据
7. 执行数据校验

## 6.3 项目字段

项目创建时需要填写：

| 字段             | 说明        |
| -------------- | --------- |
| Project Name   | 项目名称      |
| DApp Type      | DApp 类型   |
| Analysis Goal  | 本次分析目标    |
| Data Source    | 数据来源      |
| Wallet Address | 分析请求者钱包地址 |

第一版 DApp 类型可以提供：

* DEX
* GameFi
* NFT
* Generic DApp

但 Demo 的完整分析模板优先支持 DEX，其他类型可以标记为 `Coming Soon`。

## 6.4 CSV 数据字段

建议上传数据至少包含以下字段：

| 字段             | 是否必须 | 含义        |
| -------------- | ---- | --------- |
| wallet_address | 是    | 用户钱包地址    |
| event_type     | 是    | 用户行为类型    |
| amount         | 否    | 交易金额      |
| timestamp      | 是    | 行为发生时间    |
| tx_hash        | 否    | 交易哈希      |
| campaign_id    | 否    | 所属活动编号    |
| reward_amount  | 否    | 用户获得的奖励金额 |

示例：

```csv
wallet_address,event_type,amount,timestamp,tx_hash,campaign_id,reward_amount
0x123...,swap,100,2026-07-01 10:00:00,0xaaa...,campaign_01,0
0x456...,swap,25,2026-07-02 15:20:00,0xbbb...,campaign_01,0.1
0x123...,swap,80,2026-07-05 12:10:00,0xccc...,,0
```

## 6.5 数据校验

用户上传数据后，系统需要检查：

* 必填字段是否存在
* 钱包地址格式是否正确
* 时间字段能否转换
* 是否存在重复记录
* 是否存在缺失值
* 金额字段是否为数字
* 数据时间范围是否合理

系统可以返回：

```text
成功读取：1,248 条行为记录
有效钱包：326 个
异常记录：17 条
数据时间范围：2026-06-01 至 2026-07-20
```

## 6.6 为什么需要该功能

如果没有数据接入功能，平台只能展示预先写死的数据看板，无法证明产品具备实际使用能力。

加入数据接入后，完整流程变为：

```text
创建项目
→ 上传数据
→ 自动清洗
→ 执行分析
→ 返回结果
```

# 7. 功能二：数据看板与用户分层

## 7.1 功能目标

将原始行为数据转换为增长指标、用户画像和用户分层结果。

## 7.2 分析原则

核心指标必须由后端程序进行确定性计算，而不能直接让 AI 根据原始数据自由生成。

正确流程是：

```text
原始数据
→ Python 清洗和计算
→ 返回标准 JSON
→ 前端生成图表
→ AI 解释数据结果
```

这样可以保证：

* 指标口径统一
* 计算结果稳定
* 数据结果可验证
* 图表不会因为 AI 输出变化而报错
* AI 不会编造不存在的数据

## 7.3 核心增长指标

第一版建议只展示以下指标。

### 用户规模指标

* 总钱包数
* 新增钱包数
* 活跃钱包数
* 获得奖励的钱包数

### 用户行为指标

* 总交互次数
* 人均交互次数
* 总交易金额
* 人均交易金额
* 用户平均活跃天数

### 转化指标

* 首次交互用户数
* 完成第二次交互的用户数
* 二次交互率
* 完成三次以上交互的用户数

### 留存指标

* 7 日复交互率
* 14 日复交互率
* 近 7 日活跃用户数
* 流失风险用户数

### 活动指标

* 活动组用户数
* 非活动组用户数
* 活动组复交互率
* 非活动组复交互率
* 奖励总成本
* 每新增留存用户成本

## 7.4 用户增长漏斗

第一版漏斗可以设计为：

```text
全部交互用户
→ 完成首次交易
→ 完成第二次交易
→ 完成三次以上交易
→ 成为高价值用户
```

如果后续接入前端埋点，还可以扩展为：

```text
访问页面
→ 连接钱包
→ 发起签名
→ 提交交易
→ 交易成功
→ 完成二次交易
```

本次 Demo 优先使用链上行为漏斗。

## 7.5 用户分层

第一版采用规则分层，不使用复杂机器学习模型。

建议划分为五类用户。

### 7.5.1 新用户

参考规则：

```text
用户只完成过一次交互
```

业务含义：

* 已经尝试产品
* 尚未形成持续使用习惯
* 需要引导完成第二次关键行为

推荐策略：

* 新手引导
* 第二次交易任务
* 功能使用教程
* 小额手续费补贴

### 7.5.2 活跃用户

参考规则：

```text
最近 7 天交互次数不少于 3 次
```

业务含义：

* 已经形成一定使用习惯
* 对产品具有较高认知
* 是当前核心用户群体

推荐策略：

* 推送高级功能
* 会员权益
* 产品内测资格
* 社区身份认证

### 7.5.3 高价值用户

参考规则：

```text
交易金额、交易次数或手续费贡献位于前 20%
```

业务含义：

* 对平台交易量或收入贡献较高
* 需要重点维护
* 流失后影响较大

推荐策略：

* VIP 权益
* 专属手续费等级
* 新功能优先体验
* 定向访谈

### 7.5.4 流失风险用户

参考规则：

```text
历史上曾经活跃，但最近 7 天或 14 天没有继续交互
```

业务含义：

* 已经了解产品
* 但近期停止使用
* 重新激活成本可能低于获取全新用户

推荐策略：

* 召回活动
* 个性化提醒
* 限时手续费补贴
* 根据历史行为推荐功能

### 7.5.5 激励低效用户

参考规则：

```text
用户领取奖励后没有继续交互，或只完成最低任务要求
```

业务含义：

* 用户可能主要受奖励驱动
* 对产品本身的持续使用意愿较低
* 继续无差别补贴可能造成预算浪费

推荐策略：

* 不再直接发放无门槛奖励
* 提高奖励领取条件
* 使用分阶段奖励
* 观察奖励后的持续行为

## 7.6 数据看板组成

数据看板建议包含：

### 核心指标卡

* 总用户数
* 活跃用户数
* 二次交互率
* 7 日留存率
* 总交易金额
* 流失风险用户数

### 图表

* 用户增长趋势图
* 用户行为漏斗
* 用户分层分布
* 活动组和非活动组效果对比

### 用户列表

| 钱包地址     | 交易次数 | 总交易金额 | 最近交互时间     | 用户分层   |
| -------- | ---: | ----: | ---------- | ------ |
| 0x123... |    5 | 1,200 | 2026-07-19 | 活跃用户   |
| 0x456... |    1 |    30 | 2026-07-10 | 新用户    |
| 0x789... |   12 | 8,400 | 2026-07-01 | 流失风险用户 |

## 7.7 为什么使用规则分层

本次 Demo 暂时不使用：

* XGBoost
* Logistic Regression
* 流失预测模型
* 因果森林
* Uplift Model

主要原因是：

1. 缺少足够的真实历史数据
2. 缺少可靠的流失和高价值标签
3. 使用模拟数据训练模型，结果缺乏业务可信度
4. 规则分层更容易解释
5. 黑客松 Demo 更强调完整运行和产品闭环

后续在积累真实数据后，可以增加机器学习分层。

# 8. 功能三：AI 用户运营策略

## 8.1 功能目标

根据已经计算完成的结构化增长指标，生成具有数据依据的运营建议。

AI 的作用不是替代指标计算，而是：

* 解释数据结果
* 发现主要增长问题
* 确定优先运营对象
* 生成运营动作
* 推荐观察指标
* 提醒潜在风险

## 8.2 AI 输入

AI 不直接接收整份原始 CSV，而是接收后端计算完成的结构化指标。

示例：

```json
{
  "project_type": "DEX",
  "total_users": 320,
  "active_users": 62,
  "repeat_transaction_rate": 0.18,
  "retention_7d": 0.21,
  "segments": {
    "new_users": 143,
    "active_users": 62,
    "high_value_users": 24,
    "at_risk_users": 71,
    "low_efficiency_incentive_users": 20
  },
  "campaign": {
    "treatment_retention": 0.23,
    "control_retention": 0.17,
    "reward_cost": 120
  }
}
```

## 8.3 AI 输出

AI 必须返回固定 JSON，而不是自由文本。

示例：

```json
{
  "summary": "当前主要增长问题是首次交互后的二次转化率偏低。",
  "findings": [
    {
      "title": "二次交互率偏低",
      "evidence": "320 名用户中只有 18% 完成了第二次交互。",
      "priority": "high"
    }
  ],
  "strategies": [
    {
      "target_segment": "at_risk_users",
      "action": "向曾经活跃但近 7 天未交互的用户提供限时召回激励。",
      "reason": "该群体已经完成产品认知，重新激活成本可能低于重新获客。",
      "recommended_metric": "7 日重新交互率"
    }
  ],
  "risks": [
    "当前数据量有限，建议将策略作为辅助决策依据。"
  ]
}
```

## 8.4 前端展示方式

AI 输出在前端渲染为策略卡片。

每条策略需要展示：

* 目标用户
* 当前问题
* 数据证据
* 推荐动作
* 推荐原因
* 观察指标
* 风险提示

示例：

```text
目标用户：流失风险用户

问题：
该群体曾经保持活跃，但最近 7 天没有再次交互。

数据证据：
当前共有 71 名流失风险用户，占全部用户的 22.2%。

推荐动作：
向其中历史交易金额较高的用户发放限时手续费补贴。

观察指标：
7 日重新交互率。

风险：
不建议向全部流失用户发放相同奖励。
```

## 8.5 预设分析 Skill

平台后台可以设置一个固定分析 Prompt，将其理解为一个预设分析 Skill。

该 Skill 可以包含三个步骤。

### Growth Analyst

负责：

* 找出转化问题
* 找出留存问题
* 识别用户结构异常
* 判断活动效果

### Web3 Operations Strategist

负责：

* 确定目标用户
* 推荐运营动作
* 推荐激励方式
* 推荐观察指标

### Evidence Checker

负责：

* 检查结论是否有数据依据
* 避免将相关性直接解释为因果
* 检查数据量是否过少
* 检查是否存在过度补贴
* 检查羊毛用户风险

本次 Demo 不需要真正建设多个 AI Agent，可以通过一个 Prompt 完成多个分析步骤。

## 8.6 AI 失败回退

外部 AI API 可能出现：

* 免费额度耗尽
* 响应超时
* 输出格式错误
* API 临时不可用

因此需要设计本地规则回退。

示例：

```text
如果二次交互率低于 20%
→ 建议优化首次交互后的二次行为引导

如果流失风险用户占比高于 30%
→ 建议设计 7 日召回活动

如果活动组与非活动组留存差异较小
→ 建议停止无差别奖励

如果激励低效用户占比较高
→ 建议提高奖励领取条件
```

即使 AI API 失败，Demo 仍然可以输出基础运营策略。

# 9. 功能四：链上分析凭证

## 9.1 功能目标

将最终分析报告的哈希写入 Monad，为分析报告生成可验证的链上记录。

## 9.2 为什么不将完整分析放上链

以下内容不适合直接上链：

* 原始 CSV 数据
* 用户画像明细
* AI Prompt
* AI 完整输出
* 图表数据
* Python 分析过程

主要原因包括：

* 链上存储成本较高
* 数据可能包含敏感信息
* 分析计算较复杂
* 智能合约不适合执行 pandas 或 AI API
* 数据更新频率较高

因此，本项目采用：

```text
链下执行计算
+
链上保存分析凭证
```

## 9.3 为什么不让每次点击都产生交易

技术上可以让用户每次点击分析按钮都发起交易，但这种设计存在问题：

1. 每次操作都需要钱包确认
2. 用户查看和筛选数据也需要支付 Gas
3. AI 调用失败时，链上交易可能已经完成
4. 交易本身不能证明报告内容正确
5. 用户体验较差
6. 容易变成为了上链而上链

因此，平台提供两个不同操作。

### Preview Analysis

* 链下执行
* 不需要钱包签名
* 快速查看结果
* 可以重复分析
* 不产生链上记录

### Generate Verified Report

* 生成正式报告
* 计算数据哈希
* 计算报告哈希
* 请求钱包签名
* 将哈希写入 Monad
* 返回交易 Hash
* 生成可验证凭证

## 9.4 链上流程

```text
用户上传数据
→ 后端计算指标
→ AI 生成运营策略
→ 系统生成完整报告
→ 计算 datasetHash
→ 计算 reportHash
→ 用户确认钱包交易
→ 调用 Monad 智能合约
→ 写入链上分析记录
→ 返回 Analysis ID 和交易 Hash
```

## 9.5 智能合约设计

合约名称：

```text
AnalysisRegistry.sol
```

示例数据结构：

```solidity
struct AnalysisRecord {
    address requester;
    bytes32 datasetHash;
    bytes32 reportHash;
    uint256 analysisType;
    uint256 createdAt;
}
```

示例事件：

```solidity
event AnalysisRecorded(
    uint256 indexed analysisId,
    address indexed requester,
    bytes32 indexed datasetHash,
    bytes32 reportHash,
    uint256 analysisType,
    uint256 createdAt
);
```

核心函数可以包括：

```solidity
function recordAnalysis(
    bytes32 datasetHash,
    bytes32 reportHash,
    uint256 analysisType
) external returns (uint256);
```

```solidity
function getAnalysis(
    uint256 analysisId
) external view returns (AnalysisRecord memory);
```

## 9.6 链上保存内容

| 内容       | 是否上链 |
| -------- | ---- |

# 10. 完整用户流程

## 第一步：连接钱包

用户连接 MetaMask，并切换至 Monad 网络。

钱包主要用于：

* 识别分析请求者
* 签署正式分析记录
* 生成链上分析凭证

用户不连接钱包时，仍然可以使用预览分析。

## 第二步：创建项目

用户填写：

* 项目名称
* DApp 类型
* 分析目标
* 数据来源

## 第三步：上传数据

用户上传 CSV，系统完成：

* 格式检查
* 字段校验
* 数据清洗
* 异常记录提示
* 数据摘要生成

## 第四步：执行数据分析

用户点击：

```text
Analyze Data
```

系统完成：

* 用户指标计算
* 用户行为聚合
* 漏斗分析
* 留存分析
* 用户分层
* 活动效果计算

此过程全部在链下完成。

## 第五步：生成 AI 策略

用户点击：

```text
Generate Strategy
```

系统将结构化指标发送给 AI API，并返回：

* 核心结论
* 数据证据
* 用户运营策略
* 推荐指标
* 风险提示

## 第六步：生成正式报告

用户确认分析内容后，点击：

```text
Generate Verified Report
```

系统：

1. 生成最终报告
2. 计算数据哈希
3. 计算报告哈希
4. 请求钱包签名
5. 调用 Monad 合约
6. 保存链上记录
7. 返回交易 Hash

## 第七步：验证报告

用户进入验证页面，查看报告是否与链上记录一致。

> 系统发现当前二次交互率偏低，并且流失风险用户数量较高。

系统建议：

* 优先召回历史活跃用户

# 21. 最终产品闭环

```text
项目方创建项目
        ↓
上传 DApp 用户行为数据
        ↓
系统完成数据清洗和指标计算
        ↓
生成增长数据看板
        ↓
完成用户分层
        ↓
AI 生成运营策略
        ↓
项目方确认最终报告
        ↓
报告哈希写入 Monad
        ↓
生成可验证分析凭证
```

本项目的核心差异并不是单独提供一个数据看板，也不是单独调用 AI，而是将以下流程连接起来：

```text
数据接入
→ 用户分析
→ 用户分层
→ 运营决策
→ 链上验证
```

这构成了一个适合黑客松展示的最小 Web3 用户增长与数据智能 Demo。
