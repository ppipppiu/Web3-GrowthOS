# Web3 GrowthOS Mini Demo v0.1 Workflow

## 1. 文档说明

本文档用于说明 Web3 GrowthOS Mini Demo v0.1 的产品范围、用户场景、数据设计、分析规则、技术实现、开发顺序和验收标准。

本次 Mini Demo 的目标不是完成一个完整的 Web3 增长平台，而是在一周内完成一个可以本地运行、可以展示、可以解释的最小产品闭环：

```text
使用模拟 DEX 交易数据
→ 上传 CSV
→ 数据校验与清洗
→ 计算增长指标
→ 完成钱包用户分层
→ 生成规则型运营建议
→ 在前端展示分析结果
```

本版本优先验证以下核心能力：

> Web3 DApp 的交易行为数据，能否被快速转化为清晰的增长指标、钱包用户分层和可执行的基础运营建议。

---

## 2. Mini Demo 范围

### 2.1 本版本必须完成

Mini Demo v0.1 必须完成以下功能：

* 生成一份模拟 DEX 链上交易数据；
* 提供标准 CSV 数据模板；
* 支持上传 CSV 文件；
* 支持直接使用内置示例数据；
* 检查文件和字段是否合法；
* 清洗重复和无效交易记录；
* 按钱包地址聚合交易行为；
* 计算核心增长指标；
* 按固定规则完成钱包分层；
* 根据分析结果生成规则型运营建议；
* 通过单页面展示完整分析结果；
* 支持本地完整运行；
* 尽量完成公网部署。

### 2.2 本版本暂不实现

Mini Demo v0.1 暂不实现：

* Monad RPC 自动查询；
* 合约 Event Logs 自动同步；
* 第三方链上 Indexer；
* 真实项目方数据库接入；
* 链下页面访问和渠道数据；
* 钱包登录；
* 用户权限系统；
* 数据库存储；
* Gemini 或其他 AI API；
* 机器学习用户分层；
* 智能合约；
* 报告哈希上链；
* 多项目管理；
* 历史分析报告管理；
* 实时数据更新。

这些功能不是被取消，而是放入后续版本。

## 3. 项目背景

早期 Web3 DApp 通常可以获得钱包地址、交易时间、交易金额和合约交互记录，但原始交易记录不能直接回答增长问题。

项目团队通常还需要进一步了解：

* 当前有多少独立交互钱包；
* 最近有多少新增钱包；
* 最近 30 天还有多少活跃钱包；
* 有多少钱包完成了重复交易；
* 哪些钱包贡献了较高交易金额；
* 哪些历史用户可能已经流失；
* 应该优先运营哪一类钱包；
* 可以针对不同钱包采取什么运营动作。

Web3 GrowthOS 希望将原始交易数据转化为：

```text
原始交易记录
→ 数据质量结果
→ 增长指标
→ 钱包用户分层
→ 基础运营建议
```

---

## 4. Mini Demo 核心目标

本次 Mini Demo 主要验证以下产品假设：

> 即使暂时没有完整的数据团队，早期 Web3 项目也可以通过标准化的数据分析流程，将交易记录转化为基础增长洞察。

本版本完成的最小闭环为：

1. 用户上传一份符合模板要求的交易 CSV；
2. 后端检查文件类型和必填字段；
3. 后端识别重复、缺失和异常记录；
4. 后端清洗可以被处理的无效记录；
5. 后端按钱包聚合交易行为；
6. 后端计算核心增长指标；
7. 后端按照固定规则完成钱包分层；
8. 后端根据分析结果生成规则型建议；
9. 前端展示数据质量、指标、分层和建议。

## 5. Mini Demo 用户场景

### 5.1 虚拟项目背景

本次 Demo 假设服务一个虚拟的早期 DEX 项目。

| 项目   | 内容             |
| ---- | -------------- |
| 项目名称 | MonadSwap Demo |
| 项目类型 | DEX / Swap     |
| 所属生态 | Monad          |
| 数据类型 | 模拟链上交易数据       |
| 分析周期 | 过去 90 天        |
| 当前阶段 | 早期测试或冷启动阶段     |

MonadSwap 已经积累了一部分钱包交易数据，但暂时没有专职数据分析和用户增长团队。

### 5.2 项目方希望回答的问题

项目团队希望通过 Web3 GrowthOS 了解：

* 平台当前有多少独立交互钱包；
* 数据中是否存在重复或无效交易记录；
* 最近 30 天有多少活跃钱包；
* 有多少钱包完成了至少两次交易；
* 哪些钱包属于高价值钱包；
* 哪些钱包已经出现流失风险；
* 不同钱包群体可以采用什么基础运营策略。

## 6. 数据来源

### 6.1 当前版本数据来源

Mini Demo v0.1 使用合成数据进行开发、测试和展示。

合成数据用于模拟一个早期 DEX 在过去 90 天内产生的钱包交易行为。

当前版本只使用：

* 模拟链上交易数据。

当前版本不接入：

* Monad RPC；
* 真实链上历史数据；
* 真实钱包资产数据；
* 项目方链下行为数据库；
* 真实用户隐私数据。

### 6.2 为什么优先使用模拟数据

使用模拟数据的原因包括：

* 当前阶段难以获得真实项目方的完整业务数据；
* 真实链上数据不一定包含完整且适合当前分析的行为场景；
* RPC、ABI、Event Logs 和 Indexer 会显著增加开发工作；
* 模拟数据可以主动覆盖不同的钱包行为类型；
* 模拟数据可以加入重复、缺失和异常记录，用于测试数据校验；
* 当前目标是验证分析工作流，而不是构建链上数据基础设施。

### 6.3 数据真实性说明

当前数据属于 Synthetic Data，即为测试产品功能而生成的合成数据，不代表任何真实项目或真实用户。

README 和 Demo 页面中应明确说明：

> This Mini Demo uses synthetic DEX transaction data to validate the data analysis and wallet segmentation workflow. It does not contain real users, real wallet activity, or personal information.

## 7. 输入数据设计

### 7.1 CSV 文件名称

```text
sample_onchain_transactions.csv
```

### 7.2 建议数据规模

第一版建议控制为：

| 数据项    |      建议范围 |
| ------ | --------: |
| 独立钱包数量 |   100～200 |
| 交易记录数量 | 500～1,000 |
| 数据时间范围 |   过去 90 天 |
| 文件大小限制 |  不超过 2 MB |
| 行为类型   | 以 swap 为主 |

该规模足以验证分析流程，同时可以降低本地运行和部署复杂度。

### 7.3 CSV 字段

| 字段名                  | 是否必填 | 数据类型     | 含义                       |
| -------------------- | ---- | -------- | ------------------------ |
| `wallet_address`     | 是    | string   | 发起交易的钱包地址                |
| `transaction_hash`   | 是    | string   | 交易哈希                     |
| `block_time`         | 是    | datetime | 交易发生时间                   |
| `action_type`        | 是    | string   | 交易行为类型                   |
| `amount_usd`         | 是    | number   | 本次交易对应的美元金额              |
| `token_symbol`       | 否    | string   | 交易涉及的 Token              |
| `gas_fee_usd`        | 否    | number   | 本次交易的 Gas 成本             |
| `transaction_status` | 否    | string   | 交易状态，例如 success 或 failed |

### 7.4 字段约束

#### `wallet_address`

* 应以 `0x` 开头；
* 后面包含 40 位十六进制字符；
* Mini Demo 可以进行基础格式校验；
* 不要求验证该地址是否真实存在于链上。

#### `transaction_hash`

* 应以 `0x` 开头；
* 后面包含 64 位十六进制字符；
* 在同一份数据中应当唯一。

#### `block_time`

* 必须可以转换为时间类型；
* 示例格式：

```text
2026-07-20T10:20:00Z
```

#### `action_type`

Mini Demo v0.1 暂时支持：

```text
swap
add_liquidity
remove_liquidity
```

模拟数据可以以 `swap` 为主要行为。

#### `amount_usd`

* 必须能够转换为数值；
* 不允许为空；
* 不允许小于 0；
* 金额等于 0 的记录可保留，但应计入数据质量提示。

#### `transaction_status`

可选值：

```text
success
failed
```

核心指标默认只使用 `success` 交易。

如果该字段不存在，则默认所有记录均为成功交易。

### 7.5 CSV 示例

```csv
wallet_address,transaction_hash,block_time,action_type,amount_usd,token_symbol,gas_fee_usd,transaction_status
0x1111111111111111111111111111111111111111,0xaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa,2026-05-01T10:20:00Z,swap,120.00,USDC,0.02,success
0x1111111111111111111111111111111111111111,0xbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbb,2026-05-06T14:10:00Z,swap,80.00,USDC,0.02,success
0x2222222222222222222222222222222222222222,0xcccccccccccccccccccccccccccccccccccccccccccccccccccccccccccccccc,2026-07-18T09:30:00Z,swap,20.00,USDC,0.01,success
```

## 8. 分析基准时间

为了保证示例数据在未来仍然可以正确运行，本版本不直接使用程序运行当天作为分析时间。

统一使用：

```text
analysis_reference_time = 清洗后有效数据中的最大 block_time
```

例如：

```text
最大交易时间：2026-07-20
最近 7 天：2026-07-14 至 2026-07-20
最近 30 天：2026-06-21 至 2026-07-20
过去 90 天：2026-04-22 至 2026-07-20
```

所有“最近 7 天”和“最近 30 天”规则，均以 `analysis_reference_time` 为基准。

后端应在结果中返回：

```json
{
  "analysis_reference_time": "2026-07-20T23:59:59Z"
}
```

## 9. 模拟钱包行为设计

模拟数据不能完全随机生成。

应先定义钱包行为类型，再按照不同分布生成交易记录，使最终分析结果具有可解释性。

### 9.1 新钱包

行为特征：

* 首次交易发生在最近 7 天；
* 历史交易次数较少；
* 通常只有 1～2 笔交易；
* 单笔交易金额通常较低或中等。

### 9.2 活跃钱包

行为特征：

* 最近 30 天仍然发生交易；
* 最近 30 天交易次数不少于 3 次；
* 交易时间分布相对连续；
* 可能包含低金额或中金额交易。

### 9.3 高频钱包

行为特征：

* 历史交易次数明显高于普通钱包；
* 最近 30 天仍然保持交易；
* 单笔交易金额不一定很高；
* 主要用于体现交易频率价值。

高频钱包用于生成模拟数据，但 v0.1 暂不作为独立的前端分层结果。

### 9.4 高价值钱包

行为特征：

* 累计交易金额较高；
* 累计交易金额位于全部有效钱包的前 20%；
* 可能同时属于活跃钱包；
* 也可能同时属于流失风险钱包。

### 9.5 流失风险钱包

行为特征：

* 历史成功交易次数不少于 2 次；
* 过去曾经发生重复交易；
* 最近 30 天没有成功交易。

### 9.6 低频钱包

行为特征：

* 至少完成过一次成功交易；
* 历史总交易次数较少；
* 交易时间间隔较长；
* 暂时没有形成稳定重复交易习惯。

低频钱包用于形成真实的数据分布，但 v0.1 暂不作为独立的前端分层结果。

### 9.7 模拟异常记录

模拟数据还应主动加入少量异常记录，用于测试数据清洗：

* 重复交易哈希；
* 空钱包地址；
* 空交易哈希；
* 无法解析的时间；
* 金额为空；
* 金额小于 0；
* 完全空白记录；
* 失败交易；
* 金额为 0 的交易。

建议异常记录占总记录的 2%～5%。

### 9.8 建议模拟比例

以下比例仅用于生成测试数据，不代表真实 DEX 用户分布：

| 模拟行为类型  | 建议比例 |
| ------- | ---: |
| 新钱包     |  15% |
| 普通活跃钱包  |  25% |
| 高频钱包    |  10% |
| 高价值钱包   |  10% |
| 流失风险钱包  |  20% |
| 低频或其他钱包 |  20% |

部分钱包可以同时具备多个特征，因此该比例不要求严格互斥。

## 11. 数据校验流程

后端接收到 CSV 后，应依次执行以下检查。

### 11.1 文件检查

检查内容：

* 文件是否为 CSV；
* 文件是否为空；
* 文件大小是否超过 2 MB；
* CSV 是否可以被正常解析；
* 文件是否至少包含一条数据记录。

### 11.2 必填字段检查

必须包含：

```text
wallet_address
transaction_hash
block_time
action_type
amount_usd
```

缺少任意必填字段时，后端直接停止分析，并返回缺失字段名称。

返回示例：

```json
{
  "success": false,
  "error": {
    "code": "MISSING_REQUIRED_COLUMNS",
    "message": "CSV 缺少必填字段。",
    "details": {
      "missing_columns": [
        "amount_usd"
      ]
    }
  }
}
```

### 11.3 致命错误

以下情况应当停止分析：

* 文件不是 CSV；
* 文件为空；
* 文件无法解析；
* 文件超过限制；
* 缺少必填字段；
* 清洗后没有任何有效成功交易；
* 无法确定分析基准时间。

### 11.4 可清洗错误

以下情况可以删除对应记录并继续分析：

* 钱包地址为空；
* 交易哈希为空；
* 重复交易哈希；
* 时间无法解析；
* 金额为空；
* 金额无法转换为数字；
* 金额小于 0；
* 完全空白记录。

### 11.5 数据提示

以下情况不一定删除记录，但应返回提示：

* 钱包地址格式不标准；
* 交易哈希格式不标准；
* 金额等于 0；
* 可选字段缺失；
* 数据时间跨度明显少于 30 天；
* 数据中的独立钱包数过少；
* 失败交易数量较多。

### 11.6 数据清洗顺序

建议按以下顺序处理：

1. 删除完全空白记录；
2. 删除钱包地址为空的记录；
3. 删除交易哈希为空的记录；
4. 删除重复交易哈希；
5. 转换时间字段；
6. 删除时间无法解析的记录；
7. 转换金额字段；
8. 删除金额为空、无法解析或小于 0 的记录；
9. 标准化字符串字段；
10. 标准化时间为 UTC；
11. 根据交易状态区分成功和失败交易；
12. 使用成功交易进行核心分析。

### 11.7 数据质量输出

后端需要返回：

```json
{
  "data_quality": {
    "total_rows": 800,
    "valid_rows": 780,
    "invalid_rows": 20,
    "duplicate_rows": 12,
    "failed_transaction_rows": 8,
    "zero_amount_rows": 3,
    "validation_passed": true,
    "warnings": [
      "3 条成功交易的 amount_usd 为 0。",
      "当前数据的时间跨度少于 90 天。"
    ]
  }
}
```

字段说明：

* `total_rows`：原始 CSV 总行数；
* `valid_rows`：完成清洗后的有效成功交易行数；
* `invalid_rows`：因格式或字段问题被删除的行数；
* `duplicate_rows`：重复交易哈希数量；
* `failed_transaction_rows`：失败交易数量；
* `zero_amount_rows`：金额等于 0 的成功交易数量；
* `validation_passed`：是否可以继续分析；
* `warnings`：不阻断分析的数据提示。

需要满足：

```text
total_rows
=
valid_rows
+ invalid_rows
+ failed_transaction_rows
```

如果某些类别存在交叉，后端应避免重复计数。

## 12. 钱包聚合字段

完成数据清洗后，后端需要按 `wallet_address` 聚合。

每个钱包至少计算：

| 聚合字段                          | 含义            |
| ----------------------------- | ------------- |
| `first_transaction_time`      | 首次成功交易时间      |
| `last_transaction_time`       | 最近一次成功交易时间    |
| `total_transactions`          | 历史成功交易次数      |
| `transactions_7d`             | 最近 7 天成功交易次数  |
| `transactions_30d`            | 最近 30 天成功交易次数 |
| `total_volume_usd`            | 历史累计交易金额      |
| `volume_30d_usd`              | 最近 30 天交易金额   |
| `avg_transaction_amount_usd`  | 平均单笔交易金额      |
| `active_days`                 | 发生成功交易的不重复日期数 |
| `days_since_last_transaction` | 距离最近交易的天数     |

钱包聚合表是后续指标和分层的基础。

## 13. 核心增长指标

Mini Demo v0.1 优先计算以下 7 个指标。

### 13.1 独立交互钱包数

定义：

```text
清洗后有效成功交易中，不重复的钱包地址数量
```

说明：

该指标表示独立交互钱包数量，不直接等同于真实用户数量。

### 13.2 总成功交易笔数

定义：

```text
清洗后的有效成功交易记录数量
```

### 13.3 总交易金额

定义：

```text
所有有效成功交易 amount_usd 的总和
```

### 13.4 平均每个钱包交易次数

计算方式：

```text
总成功交易笔数 ÷ 独立交互钱包数
```

结果保留两位小数。

### 13.5 重复交易钱包比例

重复交易钱包定义：

```text
历史成功交易次数不少于 2 次的钱包
```

计算方式：

```text
重复交易钱包数 ÷ 独立交互钱包数
```

前端展示为百分比。

### 13.6 最近 30 天活跃钱包数

定义：

```text
在分析基准时间向前 30 天内，
至少完成过一笔成功交易的钱包数量
```

### 13.7 最近 7 天新增钱包数

定义：

```text
钱包的首次成功交易时间位于最近 7 天
```

该指标代表近期首次出现在当前数据集中的钱包，不一定等于项目真实新增用户。

### 13.8 指标输出示例

```json
{
  "metrics": {
    "unique_wallets": 150,
    "total_transactions": 780,
    "total_volume_usd": 125000.0,
    "avg_transactions_per_wallet": 5.2,
    "repeat_wallet_rate": 0.58,
    "active_wallets_30d": 85,
    "new_wallets_7d": 25
  }
}
```

## 14. 钱包分层规则

Mini Demo v0.1 使用固定、可解释的规则完成钱包分层，不使用机器学习模型。

### 14.1 新钱包

规则：

```text
first_transaction_time >= analysis_reference_time - 7 days
```

### 14.2 活跃钱包

规则：

```text
transactions_30d >= 3
```

### 14.3 高价值钱包

规则：

```text
total_volume_usd
位于全部有效钱包累计交易金额的前 20%
```

实现建议：

1. 按 `total_volume_usd` 从高到低排序；
2. 选取排名前 20% 的钱包；
3. 至少保留 1 个高价值钱包。

高价值钱包是相对标签，不代表其绝对交易金额一定很高。

### 14.4 流失风险钱包

规则：

```text
total_transactions >= 2
并且
transactions_30d = 0
```

### 14.5 标签重叠

Mini Demo v0.1 使用多标签分层。

同一个钱包可以同时属于多个标签。

例如：

```text
高价值钱包 + 流失风险钱包
```

表示该钱包历史贡献较高，但最近已经停止交易，应当被优先召回。

因为标签可以重叠：

> 各分层钱包数量之和可能大于独立钱包总数。

前端必须明确展示：

```text
Wallet segments are multi-label and may overlap.
```

### 14.6 分层输出示例

```json
{
  "segments": {
    "is_multi_label": true,
    "new_wallets": {
      "count": 25,
      "rate": 0.1667
    },
    "active_wallets": {
      "count": 45,
      "rate": 0.3
    },
    "high_value_wallets": {
      "count": 30,
      "rate": 0.2
    },
    "churn_risk_wallets": {
      "count": 35,
      "rate": 0.2333
    }
  }
}
```

## 15. 基础运营建议

Mini Demo v0.1 使用规则生成运营建议，不接入 AI API。

后端核心指标和钱包分层必须由确定性代码完成。

### 15.1 新钱包建议

触发条件：

```text
new_wallets.count > 0
```

基础建议：

> 针对最近完成首次交易的钱包，可以设计首次交易后的连续任务、第二次交易提醒或短期手续费优惠，推动其尽快形成重复交易行为。

### 15.2 重复交易不足建议

触发条件示例：

```text
repeat_wallet_rate < 0.4
```

基础建议：

> 当前重复交易钱包比例偏低，建议重点分析首次交易后的流失环节，并通过第二次交易任务、积分奖励或限时手续费优惠促进重复使用。

说明：

`0.4` 是 Demo 规则阈值，不代表行业统一标准。

### 15.3 活跃钱包建议

触发条件：

```text
active_wallets.count > 0
```

基础建议：

> 可以通过连续交易任务、等级体系、社区身份或阶段性权益，提高活跃钱包的交易频率和持续留存。

### 15.4 高价值钱包建议

触发条件：

```text
high_value_wallets.count > 0
```

基础建议：

> 建议为高价值钱包提供手续费等级、优先体验资格、专属活动或社区权益，降低核心交易用户的流失风险。

### 15.5 流失风险钱包建议

触发条件：

```text
churn_risk_wallets.count > 0
```

基础建议：

> 建议优先召回历史交易频率或金额较高、但最近 30 天未交易的钱包，并根据其历史价值设计差异化召回策略。

### 15.6 高价值流失钱包建议

触发条件：

```text
钱包同时包含 high_value 和 churn_risk 标签
```

基础建议：

> 当前存在历史贡献较高但近期停止交易的钱包，应将其设为最高优先级召回人群，并优先排查流动性、手续费、产品体验和竞品迁移等可能原因。

### 15.7 建议输出格式

```json
{
  "recommendations": [
    {
      "id": "REC_001",
      "priority": "high",
      "segment": "high_value_churn_risk",
      "title": "优先召回高价值流失钱包",
      "reason": "部分高价值钱包最近 30 天没有交易。",
      "action": "针对该人群提供差异化召回任务，并优先分析其停止交易的原因。",
      "is_rule_based": true
    }
  ]
}
```

### 15.8 AI 接入说明

Gemini API 暂不进入 Mini Demo v0.1 的必须范围。

后续版本可以采用：

```text
后端计算确定性指标和分层
→ 将结构化结果发送给 Gemini
→ Gemini 解释结果
→ Gemini 生成更完整的运营策略
```

AI 不负责：

* 直接读取原始 CSV；
* 清洗数据；
* 计算核心指标；
* 决定钱包标签；
* 替代确定性分析代码。

## 25. 风险与应对方案

### 25.1 范围持续扩大

风险：

不断加入 AI、链下数据、钱包连接和合约功能，导致基础流程无法完成。

应对：

严格按照 P0、P1、P2 执行，本周不开发 P2 功能。

### 25.2 模拟数据无法产生预期结果

风险：

完全随机的数据可能导致没有高价值用户、没有流失用户，或者所有钱包行为过于相似。

应对：

按照预设的钱包行为类型生成数据，并对最终指标进行人工检查。

### 25.3 标签数量超过总钱包数

风险：

用户误以为分层计算错误。

应对：

明确采用多标签分层，并在页面注明标签可以重叠。

### 25.4 时间窗口随日期变化

风险：

未来运行 Demo 时，所有钱包都被判断为流失。

应对：

使用数据最大交易时间作为分析基准时间。

### 25.5 公网部署失败

风险：

Render 或 Vercel 的环境、依赖或跨域配置导致部署失败。

应对：

先确保本地闭环完整，同时准备 GitHub、截图和演示视频作为保底。

### 25.6 后端返回结构频繁改变

风险：

前端开发过程中不断修改字段，导致重复返工。

应对：

在前端开发前固定 `/api/analyze` 的 JSON Schema，并使用 TypeScript 类型与后端 Pydantic 模型保持一致。
