# Web3 GrowthOS Mini Demo v0.1 Workflow

## 1. 文档说明

本文档用于说明 Web3 GrowthOS Mini Demo v0.1 的产品范围、数据来源、功能流程、分析规则、技术实现和验收标准。

本次 Mini Demo 的目标不是完成一个完整的 Web3 增长平台，而是优先验证最核心的产品闭环：

```text
上传交易数据
→ 数据校验与清洗
→ 计算增长指标
→ 完成钱包用户分层
→ 生成基础运营建议
→ 展示分析结果
```
## 2. 项目背景

早期 Web3 DApp 通常可以从链上获得大量钱包和交易记录，但原始链上数据不能直接回答以下增长问题：

当前有多少独立交互钱包？
哪些钱包是近期新增钱包？
哪些钱包仍然保持活跃？
哪些钱包贡献了较高交易金额？
哪些历史活跃钱包可能已经流失？
项目方应该优先运营哪一类钱包？

Web3 GrowthOS 希望将原始交易数据进一步转化为：
```text
增长指标
+
钱包用户分层
+
运营建议
```
3. Mini Demo 核心目标

本次 Mini Demo 主要验证以下产品假设：

Web3 DApp 的交易行为数据可以被转化为清晰的增长指标、钱包用户分层和基础运营建议。

本版本将完成以下最小闭环：

用户上传一份符合模板要求的交易 CSV。
后端检查数据字段和数据质量。
后端清洗重复或无效记录。
后端计算核心增长指标。
后端按照固定规则完成钱包用户分层。
系统根据分析结果生成基础运营建议。
前端展示数据质量、指标、分层和建议。

4. Mini Demo 用户场景

本次 Demo 假设服务一个虚拟的早期 DEX 项目：

项目名称：MonadSwap Demo
项目类型：DEX / Swap
分析周期：过去 90 天

MonadSwap 已经积累了一部分钱包交易数据，但暂时没有完整的数据分析和增长运营团队。

项目团队希望通过 Web3 GrowthOS 回答：

平台当前有多少独立交互钱包？
用户是否存在重复交易行为？
最近 30 天有多少活跃钱包？
哪些钱包属于高价值钱包？
哪些钱包已经出现流失风险？
应该针对不同钱包群体采取什么运营动作？

## 5. 数据来源
### 5.1 当前版本的数据来源
Mini Demo v0.1 优先使用合成数据进行开发和测试。
合成数据用于模拟一个早期 DEX 在过去 90 天内产生的钱包交易行为。
当前版本的数据来源包括：

模拟链上交易数据

暂时不接入：

Monad RPC 自动查询；
合约 Event Logs 自动同步；
第三方链上 Indexer；
项目方真实业务数据库；
真实用户隐私数据。

### 5.2 为什么优先使用模拟数据

使用模拟数据的原因包括：

当前阶段难以获得真实项目方的完整业务数据。
真实链上数据通常缺少渠道、活动和奖励等运营字段。
自动抓取链上数据会增加 RPC、ABI、事件解析和存储等开发工作。
模拟数据可以主动覆盖新用户、活跃用户、高价值用户和流失用户等场景。
当前最重要的目标是验证分析流程，而不是构建链上数据基础设施。

### 5.3 数据真实性说明

当前数据属于 Synthetic Data，即为测试产品功能而生成的合成数据，不代表任何真实项目或真实用户。

README 和 Demo 中应明确说明：

This Mini Demo uses synthetic DEX transaction data to validate the data analysis and wallet segmentation workflow.

## 6. 输入数据设计
### 6.1 CSV 文件名称
sample_onchain_transactions.csv
### 6.2 建议数据规模
独立钱包数量：100～200
交易记录数量：500～1,000
交易时间范围：过去 90 天
### 6.3 CSV 字段
字段名	是否必填	含义
wallet_address	是	发起交易的钱包地址
transaction_hash	是	链上交易哈希
block_time	是	交易发生时间
action_type	是	交易行为类型，例如 swap
amount_usd	是	本次交易对应的美元金额
token_symbol	否	交易涉及的 Token
gas_fee_usd	否	本次交易的 Gas 成本
### 6.4 CSV 示例
wallet_address,transaction_hash,block_time,action_type,amount_usd,token_symbol,gas_fee_usd
0xA001,0xTX001,2026-05-01 10:20:00,swap,120.00,USDC,0.02
0xA001,0xTX002,2026-05-06 14:10:00,swap,80.00,USDC,0.02
0xB001,0xTX003,2026-07-18 09:30:00,swap,20.00,USDC,0.01
## 7. 模拟用户行为设计

模拟数据不能完全随机生成，而应该先定义不同的钱包行为类型，再按照对应规则生成交易记录。

### 7.1 新用户

行为特征：

最近 7 天发生首次交易；
历史交易次数较少；
通常只有 1～2 笔交易。
### 7.2 活跃用户

行为特征：

最近 30 天仍然发生交易；
最近 30 天交易次数不少于 3 次；
交易时间分布相对连续。
### 7.3 高频用户

行为特征：

历史交易次数明显高于普通用户；
最近 30 天仍然保持交易行为；
单笔金额不一定很高。
### 7.4 高价值用户

行为特征：

累计交易金额较高；
累计交易金额位于全部钱包的前 20%；
可能同时属于活跃用户或流失风险用户。
### 7.5 流失风险用户

行为特征：

历史上至少完成过 2 笔交易；
过去曾经有一定活跃度；
最近 30 天没有任何交易。
### 7.6 低频用户

行为特征：

完成过交易；
总交易次数较少；
交易时间间隔较长；
暂时没有形成稳定的重复交易习惯。
### 7.7 标签重叠说明

不同用户标签可以存在重叠。

例如：

高价值用户
+
流失风险用户

表示该钱包历史贡献较高，但最近已经停止交易，应当被优先召回。

本版本不强制每个钱包只能属于一个分层。

## 9. 数据校验流程
后端接收到 CSV 后，需要依次执行以下检查。

### 9.1 文件检查

检查内容：

文件是否为 CSV 格式；
文件是否为空；
文件大小是否超过限制；
CSV 是否可以被正常解析。
### 9.2 字段检查

必须包含：
```
wallet_address
transaction_hash
block_time
action_type
amount_usd
```
如果缺少必填字段，后端直接返回错误信息，并指出缺少的字段名称。

### 9.3 数据质量检查

检查内容包括：
```text
钱包地址是否为空；
交易哈希是否为空；
是否存在重复交易哈希；
时间字段是否可以正常解析；
金额字段是否可以转换为数字；
金额是否小于 0；
是否存在完全空白的记录。
```
### 9.4 数据清洗

清洗规则包括：

删除重复的交易哈希；
删除钱包地址为空的记录；
删除交易哈希为空的记录；
删除时间无法解析的记录；
删除金额为空或小于 0 的记录；
将合法时间统一转换为标准时间格式；
将金额统一转换为数值类型。
### 9.5 数据质量输出

后端需要返回：
```
{
  "total_rows": 800,
  "valid_rows": 780,
  "invalid_rows": 20,
  "duplicate_rows": 12,
  "validation_passed": true
}
```

## 10. 核心增长指标
Mini Demo v0.1 优先计算以下 6 个指标。
### 10.1 独立交互钱包数
定义：
交易数据中不重复的钱包地址数量

说明：
该指标代表独立交互钱包数量，不直接等同于真实用户数量。

### 10.2 总交易笔数
定义：
完成清洗后的有效交易记录数量

### 10.3 总交易金额
定义：
所有有效交易 amount_usd 的总和

### 10.4 平均每个钱包交易次数
计算方式：
总交易笔数 ÷ 独立交互钱包数

### 10.5 重复交易钱包比例
重复交易钱包定义：
历史交易次数不少于 2 次的钱包
计算方式：
重复交易钱包数 ÷ 独立交互钱包数

### 10.6 最近 30 天活跃钱包数
定义：
在数据最大日期向前 30 天内至少发生过一笔交易的钱包数量
本版本应使用数据中的最大交易时间作为分析基准时间，而不是直接使用程序运行当天的时间。
这样可以避免样例数据过期后导致分析结果变化。

## 11. 钱包用户分层规则
### 11.1 新用户
规则：
钱包首次交易时间位于最近 7 天

### 11.2 活跃用户
规则：
最近 30 天存在交易
并且最近 30 天交易次数不少于 3 次

### 11.3 高价值用户
规则：
钱包累计交易金额位于全部钱包的前 20%

### 11.4 流失风险用户
规则：
历史总交易次数不少于 2 次
并且最近 30 天没有发生交易

### 11.5 分层结果示例
```
{
  "new_users": 25,
  "active_users": 45,
  "high_value_users": 30,
  "churn_risk_users": 35
}
```

## 12. 基础运营建议
Mini Demo v0.1 可以先使用规则生成运营建议，不强制接入 AI API。
### 12.1 新用户建议

当新用户数量较多但重复交易比例较低时：

建议优化首次交易后的产品引导，通过任务、积分或手续费优惠推动用户完成第二次交易。

### 12.2 活跃用户建议

当活跃用户规模较高时：

建议通过连续交易任务、等级体系或社区权益，提高活跃用户的使用频率和留存。

### 12.3 高价值用户建议

当高价值用户集中度较高时：

建议为高价值钱包提供 VIP 权益、手续费优惠或提前体验资格，降低核心用户流失风险。

### 12.4 流失风险用户建议

当流失风险用户数量较多时：
> 建议优先召回历史交易频率或金额较高的钱包，并针对不同流失时间设计差异化召回策略。

### 12.5 AI 接入说明
Gemini API 作为后续增强功能。
```text
后端计算确定性指标和分层
→ 将结构化结果发送给 Gemini
→ Gemini 负责解释结果和生成运营建议
```
AI 不负责直接读取原始 CSV 或计算核心指标。
